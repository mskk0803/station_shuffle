# サービスの名前

現状の案：「えきぷらっと」

# サービス概要

> どんなサービスなのかを３行で説明してください。
> 

駅・路線を選んでランダムに駅をシャッフルし、その駅へ行くとチェックイン(目的地の駅に到着したことを記録)できるアプリ。

チェックイン後は写真やコメントを投稿可能で、訪問記録を残せる。

不正な位置情報を防ぐ仕組みを搭載し、ゲストでも駅シャッフルを楽しめる。

# このサービスへの思い・作りたい理由

> このサービスの題材となるものに関してのエピソードがあれば詳しく教えてください。
このサービスを思いつくにあたって元となる思いがあれば詳しく教えてください。
> 

結婚2年目の夫婦で、電車で行ける観光スポットは一通り訪れたため、週末の外出先に悩むことが増えていた。

そんな時、「最寄り沿線の駅をシャッフルし、出た駅へ行ってみるのはどうだろう？」というアイデアが浮かび、簡単なプログラムを作成して実行。実際にその駅へ足を運んでみることにした。

降りたことのない駅で、「この駅に行こう」という発想すらなかったため、新鮮な気持ちで街を探索できた。偶然見つけた美味しいレストランや、その街ならではの特色に触れ、思いがけない発見の連続だった。

この経験から、「ほかの路線でも試したい！」という気持ちが強くなり、さらに「この仕組みを活用すれば地域の活性化にもつながるのでは？」と考えるようになった。

こうして、このサービスの開発を決意した。

# ユーザー層について

> 決めたユーザー層についてどうしてその層を対象にしたのかそれぞれ理由を教えてください。
> 
- 次の遊び先やデート先を探している層
    - 夫婦、カップル、友人同士において、旅行や遠出ではなく、手軽に「ちょっとしたお出かけ」を楽しみたい人にとって、行き先を決める手間を省きつつ、新しい発見を得られるメリットがあると考えたため。
- 電車に興味がある人
    - 途中下車を楽しむ層にとって、知らない駅で降りるということが新しい体験に繋がりやすいと考えたため。
- 日常にちょっとした刺激が欲しい人
    - 「休日がワンパターンで物足りない」と感じている人にとって、シャッフルの結果表示された駅を利用することが良い刺激になると考えたため。

# サービスの利用イメージ

> ユーザーがこのサービスをどのように利用できて、それによってどんな価値を得られるかを簡単に説明してください。
> 

## 利用方法

1. 駅や路線を選び、シャッフルボタンを押す
2. ランダムに選ばれた駅に向かい、現地でチェックイン(目的地の駅に到着したことを記録できる)
3. チェックイン後、写真やコメントを投稿できる(投稿しないことも可能)
4. フォローすると、フォロー相手が非公開ユーザーの場合、非公開ユーザの投稿を閲覧することができるようになる。また、フォロー中のユーザーがチェックインや投稿をした際に通知を受け取れるようになる。

## 得られる価値

### 新しい発見

普段下車しない駅に行くことで、新しいお店や街の魅力に出会える。

### 気軽な冒険

計画を立てる手間がなく、手軽にちょっとした冒険・旅行気分を味わえる。

### 思い出の記録

訪問した駅ごとに投稿を残し、自分だけの「旅の履歴」を作れる。

# サービスの差別化ポイント・推しポイント

> 似たようなサービスが存在する場合、そのサービスとの明確な差別化ポイントとその差別化ポイントのどこが優れているのか教えてください。
独自性の強いサービスの場合、このサービスの推しとなるポイントを教えてください。
> 

## 類似サービスとの差別化ポイント

### 駅メモ！

- 類似点
    - 駅に到着するとチェックインが可能
- 差別化ポイント
    - おでかけの「きっかけ」を提供するという点
        - 駅メモ！はゲーム性が強く、駅の収集が目的だが、本サービスは「行き先に迷ったときに新しい駅を提案し、実際に訪れて体験を楽しむ」ことが目的。
    - 偶然の出会いを楽しむという点
        - 駅メモ！は、自由に移動して収集するが、本アプリは「どこへ行くかをランダムに決める」ことで、知らない場所へ訪れるという発見を提供する。
    - 気軽に楽しめる点
        - 本アプリはゲームのような競争要素はなく、純粋にお出かけを楽しむためのツールを提供する。

## 推しポイント

「どこに行こう？」が「新しい発見へ！」。気軽にお出かけと発見を楽しもう。

# 機能候補

> 現状作ろうと思っている機能、案段階の機能をしっかりと固まっていなくても構わないのでMVPリリース時に作っていたいもの、本リリースまでに作っていたいものをそれぞれ分けて教えてください。
> 

## MVP

### 駅シャッフル機能

- 路線・駅情報の取得
- 路線・駅情報の表示・選択機能
- 選択された駅をシャッフルし一つ表示する機能

### チェックイン(駅到着記録)機能

- 位置情報取得機能
  - 「この駅に行く」ボタン押下時に取得
  - 「位置情報を更新する」ボタン押下時に取得
- チェックイン(駅到着記録)機能
- 不正な移動を検知する機能

### 投稿機能

- チェックイン(駅到着記録)後に投稿ができる機能(新規作成、編集、削除)

## 本リリース

### アカウントの公開設定機能

- 公開・非公開(フォロワーのみ閲覧可能)
  
### フォロー機能

- フォロー、フォロー解除機能
- フォローすることで、通知を受け取れるようになる機能
- 非公開アカウントをフォローした場合、非公開アカウントの投稿が閲覧可能になる機能

### 投稿機能

- 写真・コメント投稿機能

# 機能の実装方針予定

> 一般的なCRUD以外の実装予定の機能についてそれぞれどのようなイメージ(使用するAPIや)で実装する予定なのか現状考えているもので良いので教えて下さい。
> 

### 路線・駅情報の取得

[路線API](https://ekidata.jp/api/api_line.php)を使用予定。

### 路線・駅情報の表示・選択機能

SUUMOの「路線から探す」のようなUIをイメージ。

### 位置情報取得機能

[位置情報API](https://developer.mozilla.org/ja/docs/Web/API/Geolocation_API)を使用予定。

### チェックイン機能

駅の中心から半径100～300mの範囲にユーザーの位置情報が移動したときに、チェックイン(駅到着記録)を可能にする。

### 不正な移動を検知する機能

移動速度が速すぎるとき(駅に行くことを決定した瞬間の位置情報と、移動の時間が、新幹線の最高速度を上回る場合)に不正な移動であることを警告する。
